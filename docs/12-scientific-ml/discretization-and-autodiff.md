# Discretization & Automatic Differentiation

Before a neural network ever enters the picture, SciML rests on two pieces of classical
machinery: **how you turn a differential equation into something a computer can step
through** (discretization), and **how you compute the derivatives that step needs**
(finite differences, classically — automatic differentiation, in a PINN). This page walks
the numerical-integration ladder using my own ECEN 744 homework, then shows the one
conceptual swap that turns a solver into a PINN.

## The problem: a differential equation you can't just "evaluate"

A differential equation gives you the *rate of change* of a quantity, not the quantity
itself. My HW1 problem was the scalar initial-value problem

$$\frac{dy}{dt} = t - y + 2, \qquad y(0) = 2,$$

which happens to have the closed-form solution $y(t) = t + 1 + e^{-t}$ — handy, because
it lets me measure every method's error exactly. In general you don't get that luxury, so
you **discretize**: chop $[0, 1]$ into steps of size $h$ and build the solution one step
at a time from the slope the equation hands you.

## How I did it — the integrator ladder

The four methods in the homework are the standard ladder from crudest to sharpest, and
they differ only in *how many slope evaluations they average per step*.

**Forward Euler** — take one slope, at the current point, and step:

```python
def f(t, y):
    return t - y + 2          # the ODE:  dy/dt = t - y + 2

y_euler = np.zeros(n); y_euler[0] = y0
for i in range(n - 1):
    y_euler[i+1] = y_euler[i] + h * f(t[i], y_euler[i])
```

**Backward Euler** — evaluate the slope at the *next* point, which for this linear ODE I
can solve for in closed form (an *implicit* step, stable but needing an algebraic solve):

```python
for i in range(n - 1):
    y_be[i+1] = (y_be[i] + h * (t[i+1] + 2)) / (1 + h)
```

**Heun's method** — average the slope at the start and a predicted slope at the end (a
2nd-order Runge–Kutta):

```python
for i in range(n - 1):
    k1 = f(t[i], y_heun[i])
    k2 = f(t[i] + h, y_heun[i] + h * k1)
    y_heun[i+1] = y_heun[i] + (h / 2) * (k1 + k2)
```

**RK4** — the workhorse: four slope evaluations blended `1/6, 1/3, 1/3, 1/6`:

```python
for i in range(n - 1):
    xi1 = y_rk4[i]
    xi2 = y_rk4[i] + (h / 2) * f(t[i],       xi1)
    xi3 = y_rk4[i] + (h / 2) * f(t[i] + h/2, xi2)
    xi4 = y_rk4[i] + h       * f(t[i] + h/2, xi3)
    y_rk4[i+1] = y_rk4[i] + h * (
        (1/6) * f(t[i],       xi1) +
        (1/3) * f(t[i] + h/2, xi2) +
        (1/3) * f(t[i] + h/2, xi3) +
        (1/6) * f(t[i] + h,   xi4))
```

Source: `~/Projects/school/tamu-grad/sciml/sciml_hw1.py` (my ECEN 744 HW1).

The payoff is **order of accuracy**: halve the step $h$ and Euler's error halves (1st
order), Heun's error quarters (2nd order), and RK4's error drops by 16× (4th order). My
homework's error-vs-$h$ table makes this concrete — at $t = 1$:

| $h$ | Euler | Backward Euler | Heun | RK4 |
|---|---|---|---|---|
| 0.10 | 1.6 × 10⁻² | 1.9 × 10⁻² | 6.1 × 10⁻⁴ | 1.5 × 10⁻⁷ |
| 0.05 | 8.2 × 10⁻³ | 9.1 × 10⁻³ | 1.6 × 10⁻⁴ | 9.8 × 10⁻⁹ |
| 0.01 | 1.7 × 10⁻³ | 1.8 × 10⁻³ | 6.2 × 10⁻⁶ | 1.6 × 10⁻¹¹ |

(Values regenerated by re-running the script above; on a log–log plot each column is a
straight line whose slope *is* the method's order.)

This same ETDRK4 spectral idea — a high-order integrator marching a PDE forward on a fine
grid — is exactly how I generate the **ground-truth reference** for the hardest benchmark
in my final project, the Kuramoto–Sivashinsky equation:

```python
# ETDRK4 (Kassam & Trefethen, 2005): exact treatment of the stiff linear part,
# 4th-order Runge-Kutta-like treatment of the nonlinear part, in Fourier space.
for step in range(1, n_steps + 1):
    Nv = nonlinear(v)
    a  = E2 * v + Q * Nv
    Na = nonlinear(a)
    b  = E2 * v + Q * Na
    Nb = nonlinear(b)
    c  = E2 * a + Q * (2.0 * Nb - Nv)
    Nc = nonlinear(c)
    v  = E * v + Nv * f1 + 2.0 * (Na + Nb) * f2 + Nc * f3
```

(my own reference-data generator).

## The swap that makes a PINN: finite differences → autodiff

Every method above needs the equation's derivatives. A classical solver approximates them
with **finite differences** on the grid — `(y[i+1] - y[i]) / h` and its higher-order
cousins — which is why it needs a mesh, and why the error is tied to $h$.

A PINN throws the mesh away. It represents the solution as a network $u_\theta(x, t)$ and
computes the derivatives it needs — $u_t$, $u_x$, $u_{xx}$ — by **automatic
differentiation of the network with respect to its own inputs**. These are *exact*
(to floating point), not $O(h)$ approximations, and they can be evaluated at *any* point,
not just grid nodes. Concretely, in JAX a second spatial derivative is just two nested
`grad`s:

```python
u_x  = jax.grad(u_scalar, argnums=1)          # d u / d x
u_xx = jax.grad(u_x,       argnums=1)          # d²u / d x²
```

That is the entire conceptual bridge. Automatic differentiation is the same tool that
backprop uses to get `∂loss/∂weights`; a PINN simply *also* points it at the inputs to
assemble the PDE residual. The [next page](pinns.md) builds a full residual loss out of
exactly these derivatives.

> **Full derivation.** The autodiff mechanics (forward vs. reverse mode, the computational
> graph) are developed for the weights in
> [Neural Network Fundamentals](../08-machine-learning/deep-learning/neural-network-fundamentals.md);
> the only new thing here is differentiating with respect to inputs.

## Gotchas

- **Backward Euler needed an algebraic solve.** For this linear ODE I could rearrange the
  implicit update by hand; for a nonlinear ODE you'd need a Newton solve *inside* each
  step. Implicit methods buy stability (big steps on stiff problems) at the price of that
  inner solve.
- **Higher order ≠ always better.** RK4's $10^{-11}$ error at $h = 0.01$ is past the point
  where rounding matters; on a stiff problem a stable low-order implicit method can beat a
  high-order explicit one that has to take tiny steps.
- **Autodiff derivatives are exact but not free.** Each extra spatial derivative in the
  residual is another pass through the graph — a 4th-order PDE like Kuramoto–Sivashinsky
  needs `u_xxxx`, i.e. four nested `grad`s per collocation point, which is a real cost in
  the training loop.
